# Self-Hosting Archie — Customer Notes

> What a community operator needs to know to run Archie on their own server.
> This document describes the configuration model that makes Archie deployable
> without editing source code.

---

## Configuration model

Archie reads all deployer-specific values from environment variables. You do
**not** edit source code to point it at your server.

The template is `dashboard/.env.example`. Copy it, fill in your values, and
Archie builds every client config, subscription URL, and TLS setting from
those values.

```bash
cp dashboard/.env.example dashboard/.env.local   # local dev
# or put the same vars in your docker-compose env_file for production
```

### Values you must set for a fresh install

| Variable | What it is | How to get it |
|---|---|---|
| `NEXT_PUBLIC_SERVER_IP` / `SERVER_IP` | Public IP clients dial for direct protocols | your VPS public IP |
| `NEXT_PUBLIC_SERVER_DOMAIN` / `SERVER_DOMAIN` | Domain fronting CDN/TLS protocols | your domain (e.g. `vpn.yourdomain.com`) |
| `NEXT_PUBLIC_SERVER_PORT` / `SERVER_PORT` | Port for CDN/TLS protocols (usually 443) | `443` behind Cloudflare/nginx |
| `NEXT_PUBLIC_PUBLIC_BASE_URL` | Full dashboard URL incl. basePath | `https://vpn.yourdomain.com/v3` |
| `NEXT_PUBLIC_VLESS_PBK` / `VLESS_PBK` | VLESS Reality public key | `xray x25519` → take the public key |
| `NEXT_PUBLIC_VLESS_SID` / `VLESS_SID` | VLESS Reality short ID | `openssl rand -hex 8` |
| `NEXT_PUBLIC_VLESS_SNI` / `VLESS_SNI` | Reality camouflage host | a popular TLS site, e.g. `www.cloudflare.com` |
| `AUTH_SECRET` | HMAC secret for operator session cookies | `openssl rand -base64 32` |
| `API_AUTH_TOKEN` | Bearer token the Python API requires on mutating routes | `openssl rand -hex 32` |

> **The Reality keypair is critical.** Generate your own with `xray x25519`.
> Never reuse the values from the reference deployment — they are public and
> reusing them compromises your Reality camouflage.

### Values you may want to set

| Variable | What it is | Default |
|---|---|---|
| `NEXT_PUBLIC_BRAND_NAME` / `BRAND_NAME` | Product name in tab title + subscription labels | `My VPN` (placeholder) |
| `ABUSEIPDB_API_KEY` | IP reputation enrichment (optional) | empty (enrichment disabled) |
| `NEXT_PUBLIC_BASE_PATH` | basePath the dashboard is served under | `/v3` |

## What flows through these values

Once set, every downstream surface is built from them automatically:

- **VPN URIs** (VLESS, VMess, Trojan, SS, Hysteria2, WireGuard) embed your
  server IP/domain/Reality keys — generated by `lib/xray-config.ts`.
- **Subscription URLs** (`/api/sub/<token>`, `/api/qr/<email>`) point at your
  dashboard base URL — built from `publicBaseUrl`.
- **Inbound templates** (the Inbounds page create/edit flow) stamp your domain
  into `tlsSettings.serverName`, `wsSettings.headers.Host`, etc. — via
  `lib/inbound-config.ts`.
- **Subscription profile titles and the browser tab** use your brand name.

A single value change in `.env.local` / compose propagates everywhere. There is
no second place to update.

## Domains & SNI — Reality vs CDN protocols

Archie uses **two different domains depending on the protocol**, and they serve
opposite purposes. Getting this right is what keeps the service hard to detect.

| Protocol | SNI / host it uses | Set by | Why |
|---|---|---|---|
| **VLESS Reality** (direct, no domain/cert) | a **decoy** domain — e.g. `www.cloudflare.com` | `VLESS_SNI` | Reality borrows a real, popular site's TLS handshake. To a censor the connection looks like normal traffic to that site, not to your server. |
| **WS / gRPC / Trojan-TLS / XHTTP** (TLS + your domain) | your **real** domain — e.g. `vpn.yourdomain.com` | `SERVER_DOMAIN` | The TLS certificate is issued for *your* domain. Whatever terminates that TLS — Cloudflare **or** your own nginx — routes by the SNI, so it must be your real domain. A decoy never reaches your server. |

The split is not really "CDN vs not" — it's **"does the protocol present a TLS cert
for a domain?"** Reality doesn't (it fakes someone else's), so it uses a decoy.
Everything else does, so it uses your real domain — regardless of whether
Cloudflare is in front.

**Rules of thumb**

- **Reality → decoy domain.** Pick a high-reputation TLS site that is *not* blocked
  in your users' region (`www.cloudflare.com`, `www.apple.com`, etc.). It must be a
  site that genuinely serves TLS 1.3 **and still works as a decoy** — some big sites
  stop cooperating over time (`www.microsoft.com` did; the default is now
  `www.cloudflare.com`). The decoy in `VLESS_SNI` must match the `serverNames` of
  your Reality inbound — Archie keeps the share-links and the server config in sync
  from this one value.
- **TLS-with-domain protocols → your real domain.** Never put a decoy in
  `tlsSettings.serverName` or `wsSettings.headers.Host` for these — the TLS endpoint
  routes by it, so a decoy breaks the connection.
- The **subscription URL** (`/api/sub/…`) uses your dashboard domain. That's just
  the endpoint the client fetches to get its config; it is not part of the VPN
  handshake and is not a detection surface.

> In the **Inbounds → edit inbound** screen, the **SNI** quick-field shows a hint
> reminding you which domain applies: a decoy for Reality, your real domain for
> the TLS-with-domain protocols.

**Troubleshooting: "VLESS connects but I get no internet"**

This is almost always the **Reality decoy (SNI)**, not your keys, network, or
firewall. When the decoy site stops cooperating (e.g. `www.microsoft.com`
stopped working), the Reality handshake still *completes* against that site and
the client shows **connected** — but no traffic is ever proxied, and **nothing
logs an error**. It just looks like "no internet".

The dashboard's **Server Health** card flags this directly
(`VLESS_SNI_DEAD_DECOY`) when it detects a known-dead decoy.

To fix it:

1. Set `VLESS_SNI` / `NEXT_PUBLIC_VLESS_SNI` to a working decoy
   (`www.cloudflare.com` is the current default) and restart Xray + rebuild the
   dashboard so the server `serverNames` and the share-links agree.
2. **You do not need to reissue keys.** The key (its UUID) is unchanged — only
   the `sni=` inside its share-link changes. Just have each user re-import the
   **updated link** for their existing key. Creating brand-new keys is wasted
   effort here.

Before blaming the decoy, rule out the obvious: the VPN ports (443 for VLESS,
2096 for Hysteria2, 51820 for WireGuard) must be reachable from your users'
networks — your cloud provider's firewall / security group has to allow them
inbound, not just from your own IP.

### Is Cloudflare a dependency?

**No — Cloudflare is optional.** What you get out of the box without it:

| Protocol | Works without Cloudflare? | Needs |
|---|---|---|
| **VLESS Reality** | ✅ yes | nothing but your VPS IP — no domain, no cert |
| **Hysteria2** | ✅ yes | direct UDP, its own TLS |
| **WireGuard** | ✅ yes | direct UDP, kernel crypto |
| **WS / gRPC / Trojan-TLS / XHTTP** | ⚠️ only with a **publicly-trusted cert** | a domain pointed at your VPS **+** a Let's Encrypt cert on nginx |

So a fully functional VPN with **no Cloudflare** is **Reality + Hysteria2 +
WireGuard** — exactly the **Performance** preset. All three connect straight to
your server IP.

The CDN-style protocols (WS/gRPC/Trojan-TLS/XHTTP) are the only ones that lean on
a fronting layer. In the reference deployment they sit behind Cloudflare and the
origin uses a **Cloudflare Origin certificate** — which is *not* trusted by client
devices directly (that's why the direct Trojan-TLS link carries `allowInsecure=1`).
To run those protocols **without** Cloudflare you must:

1. Point the domain's DNS record straight at your VPS, and
2. Install a **publicly-trusted certificate** (e.g. Let's Encrypt) on nginx.

Then they work direct, with your real domain as the SNI. The trade-off: your real
IP is exposed via DNS, you lose Cloudflare's DDoS protection, and the traffic no
longer hides as ordinary CDN traffic. If you want CDN camouflage and IP hiding,
keep Cloudflare in front of those protocols.

> **Bottom line:** Cloudflare is a dependency *only* for the CDN-fronted protocols
> as shipped. Reality + Hysteria2 + WireGuard give a complete, hard-to-detect VPN
> with zero CDN. The guided setup wizard asks which mode you want and configures
> certs accordingly.

## First-run setup

Archie includes a first-user setup flow: the first time you open the dashboard
with no auth users configured, you're prompted to create the initial `owner`
account. Once an account exists, setup closes automatically.

This means **there is no default password** — you create your own on first run.

---

## Reference deployment note

The reference deployment ships with built-in fallback values for the
deployer-specific vars so it keeps working without env vars set. For your own
install, **do not rely on those fallbacks** — they point at the reference
server. Set your own values as described above.

Removing the fallbacks from `lib/server-config.ts` and `api/vpn-api.py` is a
cleanup step the reference deployment owner performs once env vars are set in
their compose file. It does not affect your install either way.
